## 1. 触发描述写作模式

### 模式：功能+情境+关键词三段式

```yaml
description: |
  [核心功能一句话]。
  当用户[具体情境描述]时触发。即使用户没有明确说出技能名称，只要
  提到[关键词列表]，也应该使用此技能。
```

**示例：**
```yaml
description: |
  将 Excel/CSV 数据转化为交互式可视化图表和分析报告。
  当用户提到数据分析、图表绘制、销售数据、趋势分析、Excel 文件处理时触发。
  即使用户只说"帮我看看这份数据"并附上了表格文件，也应该使用此技能。
```

### 模式：对抗 undertrigger 的主动描述

如果担心技能触发不足，在描述结尾加一句明确的触发邀请：
```yaml
description: |
  ...
  只要用户的请求涉及[领域]，即使措辞模糊，也请考虑使用此技能。
```

## 2. 指令写作模式

### 模式：解释"为什么"而非只说"做什么"

❌ 不好：
```
ALWAYS add error handling to every function.
```

✅ 好：
```
为每个函数添加错误处理。这样做是因为该技能生成的代码
会在生产环境直接运行——未处理的异常会导致用户数据丢失，
而良好的错误信息能让用户快速定位问题。
```

### 模式：条件指令

当行为依赖上下文时，使用清晰的条件结构：
```markdown
## 输出格式选择
- 如果用户明确指定了格式（JSON、CSV、Markdown）→ 使用指定格式
- 如果输出将被其他程序处理 → 使用 JSON
- 如果用户是非技术人员 → 使用 Markdown 表格
- 默认情况下 → 使用 Markdown
```

### 模式：决策树

对于复杂的多路分支逻辑：
```markdown
## 处理流程
1. 检查输入文件类型
   - .xlsx / .csv → 走数据提取流程（见下文）
   - .pdf → 走 OCR 流程（见 references/ocr-guide.md）
   - 其他 → 告知用户支持的格式，不要猜测
2. 验证数据完整性...
```

## 3. 输出格式定义模式

### 模式：精确模板

```markdown
## 报告结构
始终使用以下 Markdown 模板（不要改变标题层级）：

# [技能名称] 分析报告
**生成时间**：[ISO 8601 时间戳]

## 执行摘要
[2-3 句话总结核心发现]

## 详细分析
[按发现的重要性降序排列]

## 建议行动
[编号列表，最重要的放在最前面]
```

### 模式：示例驱动

```markdown
## 提交信息格式

**格式**：`<类型>(<范围>): <描述>`

**示例 1**：
输入："添加了用户认证功能，用了 JWT"
输出：`feat(auth): 实现基于 JWT 的用户认证`

**示例 2**：
输入："修了一个登录时崩溃的 bug"
输出：`fix(auth): 修复登录时 NullPointerException 崩溃`
```

## 4. 错误处理模式

### 模式：明确的失败路径

```markdown
## 当出现以下情况时

**输入文件不存在**
→ 告知用户文件路径，询问是否重新提供，不要继续执行

**必需字段缺失**
→ 列出所有缺失字段，询问用户补充，不要用默认值静默跳过

**外部 API 调用失败**
→ 显示具体的错误码和消息，建议用户检查网络/凭据，
   提供一个不依赖 API 的降级输出（如果有意义的话）
```

## 6. 多环境适配模式

当技能依赖特定工具时，提供降级路径：

```markdown
## 工具依赖

**首选**：使用 `pandoc` 进行文档转换（质量最高）

**降级**：如果 pandoc 不可用：
- 检查是否可以通过 `pip install pypandoc` 获取
- 若仍不可用，使用 Python 的 `python-docx` 库（功能受限，告知用户）
- 若所有方式都不可用，生成 Markdown 文件并说明无法转为 Word 格式

永远不要静默失败——如果降级，告诉用户发生了什么以及原因。
```

## 7. 常见反模式（要避免的）

### ❌ 过度规范（Overspecification）
```
ALWAYS generate exactly 5 bullet points. NEVER use more than 100 words per section.
MUST include exactly 3 examples.
```
问题：这类僵化规则无法适应不同任务的实际需求，会让输出变得生硬。

✅ 替代：解释你希望输出简洁的原因，让模型根据内容量判断。

---

### ❌ 悬空引用（Dangling References）
```
详细配置见 references/config.md。
```
但 `references/config.md` 并不存在。

✅ 替代：只引用实际存在的文件，或明确标注"（待创建）"。

---

### ❌ 自相矛盾（Internal Contradiction）
```
始终先生成摘要再写详细内容。
...（后文）...
先收集所有内容，最后生成摘要。
```

✅ 替代：写完草稿后重新通读一遍，检查是否前后一致。

---

### ❌ 无上下文的 MUST（Uncontextualized MUST）
```
MUST validate all inputs before processing.
```
问题：没有告诉模型怎么验证、验证什么、验证失败时怎么做。

✅ 替代：
```
在处理之前验证输入。检查：文件是否存在、格式是否为支持的类型（见上方列表）、
必需字段是否都有值。如果验证失败，列出具体失败原因并停止，不要猜测用户意图。
```
