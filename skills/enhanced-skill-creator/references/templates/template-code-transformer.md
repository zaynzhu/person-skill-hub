## 转换前分析

在开始转换之前，先分析源代码并告知用户：

1. **转换复杂度评估**
   - 简单（主要是语法差异）：直接转换
   - 中等（API/模式差异）：列出需要手动决策的地方
   - 复杂（架构差异）：建议拆分为多个步骤，先讨论方案

2. **已知差异点**
   列出源技术和目标技术之间的主要差异，以及将如何处理：
   ```
   - [源特性 A] → [目标等价物 A]（直接替换）
   - [源特性 B] → [目标等价物 B]（需要调整，因为...）
   - [源特性 C] → 目标无直接等价物（建议使用 [替代方案]）
   ```

3. **无法自动处理的部分**
   提前告知哪些需要人工决策（不要静默做假设）。

## 转换流程

1. **逐文件/逐模块转换**：按依赖顺序处理（底层模块先转换）
2. **保留原始文件**：将转换结果写到新路径，不覆盖原文件
3. **生成差异报告**：说明每个文件发生了哪些重要变化
4. **测试建议**：给出转换后的验证步骤

---

## 输出

- 转换后的文件保存到 `[目标路径]`
- 生成 `MIGRATION_NOTES.md` 记录：所有不确定的转换决策、需要人工验证的地方、已知的行为差异

---

## 转换后的必要操作

告知用户完成转换后需要手动做的事情（永远不要假装转换是完美无缺的）：
1. 安装目标技术栈的依赖
2. 运行测试套件验证功能一致性
3. 检查 `MIGRATION_NOTES.md` 中标注的需要人工确认的地方
